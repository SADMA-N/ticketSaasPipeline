## Full Working Structure

```
POST /tickets
  │
  ├─ controller: Zod validate
  ├─ service: createTask() → Postgres { state: "pending" }
  ├─ service: SQS.sendMessage({ taskId }) → LocalStack queue   [AC2]
  └─ controller: return 202 immediately                         [AC1]
       ↑ API returns here — no waiting for AI

      ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ separate process ─ ─ ─

Worker (npm run dev:worker)                                     [AC3]
  │
  worker.ts: poll() loop
    └─ receiveMessages() → SQS long-poll (20s wait)
         └─ for each message:
              │  parse { taskId } from msg.Body
              │
              ├─ processJob(taskId, receiptHandle)
              │    ├─ getTask(taskId) from Postgres
              │    ├─ GUARD: terminal state → deleteMessage + return
              │    ├─ updateTask({ state: "processing",         [AC4]
              │    │               currentPhase: "phase_1",
              │    │               stateChangedAt: new Date() })
              │    ├─ Phase 1: updateTask({ phase1Retries: +1 })
              │    │           TODO: runPhase1() ← Epic 3
              │    └─ Phase 2: updateTask({ phase2Retries: +1 })
              │                TODO: runPhase2() ← Epic 4
              │
              └─ deleteMessage(receiptHandle)  ← only on success
                   ✗ job throws → no delete → visibility timeout
                                → SQS auto-retries
```

## SQS Retry Behaviour (Built-in)

```
Worker throws (job fails)
  └─ No deleteMessage called
       └─ Visibility timeout expires
            └─ SQS makes message visible again
                 └─ Worker picks up and retries

maxReceiveCount reached (3)
  └─ SQS moves to Dead Letter Queue (DLQ)
       └─ DLQ listener (Epic 5):
            !phase1Done → state: "needs_manual_review"
             phase1Done → state: "completed_with_fallback"
```

## What Was Added After US-2.1 (Checklist Items 5 + 6)

### Item 5 — Phase Retry Limit (MAX_PHASE_RETRIES)

```
PROBLEM BEFORE:
  phase1Retries counter incremented on every SQS delivery
  but nothing ever CHECKED it → task could retry forever

FIX:
  src/worker/processor.ts
    const MAX_PHASE_RETRIES = 3

  processJob() flow with retry limit:
  ─────────────────────────────────────────────────────
  SQS delivers message (attempt N)
    │
    ├── getTask(taskId) → task.phase1Retries = N-1
    │
    ├── if (!task.phase1Done)
    │     │
    │     ├── CHECK: task.phase1Retries >= MAX_PHASE_RETRIES (3)?
    │     │     │
    │     │     ✓ YES (4th delivery, retries already = 3)
    │     │     │   updateTask({ state: "needs_manual_review",
    │     │     │                currentPhase: null })
    │     │     │   deleteMessage()   ← remove from queue cleanly
    │     │     │   return            ← stop processing
    │     │     │
    │     │     ✗ NO (retries < 3)
    │     │         updateTask({ phase1Retries: +1 })
    │     │         runPhase1()  ← attempt the AI call
    │     │         ✗ throws → no deleteMessage → SQS retries again
    │     │         ✓ updateTask({ phase1Done: true, currentPhase: "phase_2" })


TIMELINE EXAMPLE (phase1 always fails):
──────────────────────────────────────
  Delivery 1: phase1Retries=0 → NOT >= 3 → increment to 1 → runPhase1 throws
  Delivery 2: phase1Retries=1 → NOT >= 3 → increment to 2 → runPhase1 throws
  Delivery 3: phase1Retries=2 → NOT >= 3 → increment to 3 → runPhase1 throws
  Delivery 4: phase1Retries=3 → YES >= 3 → needs_manual_review → deleteMessage


STATE RESULT:
  task.state        = "needs_manual_review"
  task.currentPhase = null
  task.phase1Retries = 3
```

### Item 6 — DLQ Infrastructure

```
WHAT IS A DLQ?
──────────────
Dead Letter Queue = a second SQS queue that catches messages
SQS automatically moves there when maxReceiveCount exceeded.

Without it: failed messages stay in main queue forever.
With it:    failed messages go to DLQ → our handler cleans up.


HOW IT'S SET UP (scripts/setup-queues.sh):
────────────────────────────────────────────
npm run dev:queues
  │
  ├── Delete old queues (if exist)
  ├── Create ticket-queue-dlq          ← the DLQ
  ├── Get DLQ ARN
  └── Create ticket-queue with:
        RedrivePolicy: {
          deadLetterTargetArn: <DLQ ARN>,
          maxReceiveCount: 3             ← after 3 failed deliveries → DLQ
        }


TWO QUEUES IN LOCALSTACK:
──────────────────────────
  ticket-queue          ← main queue (worker polls this)
  ticket-queue-dlq      ← dead letter queue (dlq poller polls this)


WORKER RUNS TWO POLLERS IN PARALLEL (worker.ts):
──────────────────────────────────────────────────
  Promise.all([poll(), pollDlq()])
       ________|           |
       │                   └── pollDlq()
       │                         └── receiveDlqMessages()   ← queue.ts
       │                               └── SQS long-poll on ticket-queue-dlq
       │                                     for each msg:
       │                                       handleDlqMessage(taskId, receipt)
       │                                         └── dlqHandler.ts
       │
       └── poll()
             └── receiveMessages()   ← queue.ts
                   └── SQS long-poll on ticket-queue


DLQ HANDLER LOGIC (src/worker/dlqHandler.ts):
──────────────────────────────────────────────
  handleDlqMessage(taskId, receiptHandle)
    │
    ├── getTask(taskId) from Postgres
    │
    ├── task.phase1Done?
    │     ✓ YES → state: "completed_with_fallback"
    │              (Phase 1 succeeded, Phase 2 failed repeatedly)
    │     ✗ NO  → state: "needs_manual_review"
    │              (Phase 1 itself failed repeatedly)
    │
    ├── updateTask({ state, currentPhase: null, stateChangedAt })
    └── deleteDlqMessage(receiptHandle)   ← removes from DLQ


FULL RETRY + DLQ FLOW:
───────────────────────
  POST /tickets → [pending]
       │
  Worker picks up → [processing]
       │
  runPhase1() fails
       │
  SQS visibility timeout → message becomes visible again
       │
  Worker retries (delivery 2) → fails again
       │
  Worker retries (delivery 3) → fails again
       │
  maxReceiveCount (3) reached
       │
  SQS moves message → ticket-queue-dlq
       │
  pollDlq() picks it up → handleDlqMessage()
       │
  phase1Done=false → state: "needs_manual_review"  ✅ never stuck, never "failed"


NOTE: MAX_PHASE_RETRIES (Item 5) catches the case BEFORE DLQ.
      DLQ catches unexpected crashes/unhandled errors our code missed.
      Both work together as two layers of protection.


NEW FILES:
──────────
  scripts/setup-queues.sh      ← creates both queues (run after docker up)
  src/worker/dlqHandler.ts     ← handleDlqMessage()
  src/worker/workerEvents.ts   ← EventEmitter singleton

MODIFIED FILES:
───────────────
  src/worker/queue.ts          ← added receiveDlqMessages(), deleteDlqMessage()
  src/worker/worker.ts         ← added pollDlq(), Promise.all([poll(), pollDlq()])
  src/worker/processor.ts      ← added MAX_PHASE_RETRIES check before Phase 1
  src/config/env.ts            ← added SQS_DLQ_URL
  package.json                 ← added "dev:queues" script
```
