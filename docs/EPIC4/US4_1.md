# US-4.1 — Phase 2 AI Resolution Draft (Epic 4)

## Context

US-3.1 complete. `processor.ts` had Phase 2 TODO stub. Now replacing stub with
Portkey → Gemini call that generates resolution draft, internal note, and next
actions using Phase 1 triage as context. Integrates with existing retry/state
machine.

## AC Map

| AC | What it means in code |
|----|----------------------|
| AC1 | `runPhase2()` returns all 3 fields: `response_draft`, `internal_note`, `next_actions` |
| AC2 | Phase 1 output passed into prompt — draft not generated from raw ticket alone |
| AC3 | Output is Zod-validated structured object, stored in task record |
| AC4 | Missing fields / parse failure → `throw` → SQS retry (existing E2 mechanism) |
| AC5 | `response_draft` prompted to start with `[DRAFT]` — no auto-send mechanism exists |
| AC6 | `buildOutputs()` returns `{ phase_1, phase_2 }` when `state === "completed"` |

---

## US-4.1 — How Each AC Connects in the Codebase

```
AC1: runPhase2() returns all 3 fields
──────────────────────────────────────
src/worker/processor.ts: processJob()
  └── runPhase2(freshTask.inputTicket, freshTask.phase1Output)
        └── src/worker/phase2.ts
              └── callLLM(ticket, triage)
                    └── portkey.chat.completions.create({
                          tools: [{ function: { name: "generate_resolution",
                            parameters: {
                              response_draft, internal_note, next_actions
                            }
                          }}],
                          tool_choice: { type: "function", name: "generate_resolution" }
                        })
                        → src/config/portkey.ts (Portkey client)
                        → Portkey gateway (portkey.ai)
                        → Gemini API (gemini-2.0-flash) [fallback: Groq]
                        ← tool_calls[0].function.arguments (JSON string)
                        → Phase2OutputSchema.parse()
                        ← { response_draft, internal_note, next_actions }   ✅ AC1


AC2: Phase 1 output used as context in Phase 2 prompt
──────────────────────────────────────────────────────
src/worker/phase2.ts: callLLM(ticket, triage)
  └── messages: [
        { role: "system", content: "You are a support agent assistant..." },
        { role: "user", content:
            "Ticket:\n
             Subject: ${ticket.subject}\n
             Body: ${ticket.body}\n\n
             Triage:\n                         ← Phase 1 output injected here
             Category: ${triage.category}\n

        }
      ]
      ↑ Both raw ticket AND triage (Phase 1 result) in the same prompt

AC3: Output structured + persisted to task record
──────────────────────────────────────────────────
src/worker/phase2.ts
  └── Phase2OutputSchema = z.object({
        response_draft: z.string(),
        internal_note:  z.string(),

      })
      tool_choice forces function calling → guaranteed JSON schema
      Zod validates before returning

src/worker/processor.ts: processJob()
  └── updateTask(taskId, {
        phase2Output: phase2Output,   ← JSON stored in Postgres
        phase2Done: true,             ← checkpoint flag set
        state: "completed",           ← final state
        currentPhase: null,
      })
      → prisma.task.update()          ← persisted after validation          ✅ AC3


AC4: Parse failure → throw → SQS retry
────────────────────────────────────────

  ├── !choices?.length → throw Phase2Error("Empty choices in LLM response")

  └── Phase2OutputSchema.parse(JSON.parse(...))
        ↑ .parse() throws ZodError if any field missing/wrong type     ← AC4 trigger

src/worker/worker.ts

      catch → no deleteMessage called
        → visibility timeout expires
        → SQS makes message visible again
        → worker retries                                                     ✅ AC4
```
