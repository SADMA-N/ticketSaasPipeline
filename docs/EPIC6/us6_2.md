# US-6.2 — Structured Audit Logging

## 1. Big Picture

Every time a ticket moves through the pipeline, we now **write a log entry** — a structured JSON record saved to stdout. Each log always has `task_id`, `event`, `timestamp`, and optional fields like `phase`, `outcome`, `duration_ms`. This creates a full audit trail so anyone can grep by `task_id` and see exactly what happened to a ticket, in order.

---

## 2. Step-by-Step Flow

```
ticket submitted
      ↓
worker picks it up → calls processJob()
      ↓
processJob calls emitSocketEvent() for each step
      ↓
emitSocketEvent builds payload → logger.info() writes JSON log → sends socket event
      ↓
terminal event? → duration_ms included in metadata
```

---

## 3. Line-by-Line (important parts only)

**`emitter.ts`**
```ts
const phase = (metadata.phase as string | undefined) ?? EVENT_PHASE[event];
```
Check if caller already passed `phase` in metadata (e.g. retry does). If not, look it up from `EVENT_PHASE` map. `??` means "use right side only if left is null/undefined".

```ts
const outcome = TERMINAL_EVENTS.has(event) ? event : undefined;
```
If event is a final state (`completed`, `needs_manual_review`, etc.) → set `outcome` to event name. Otherwise undefined — won't appear in log.

```ts
logger.info(
  {
    ...payload,
    ...(phase !== undefined && { phase }),
    ...(outcome !== undefined && { outcome }),
  },
  "socket event",
);
```
`...payload` spreads `task_id`, `event`, `timestamp`, `metadata` into the log object. The `&&` trick only adds `phase`/`outcome` fields when they exist — so they never appear as `null` in logs where they don't apply.

**`processor.ts`**
```ts
emitSocketEvent(taskId, "completed", {
  duration_ms: Date.now() - freshTask.createdAt.getTime(),
});
```
`Date.now()` = now in milliseconds. `createdAt.getTime()` = when ticket was first saved. Subtract = how long the whole thing took.

---

## 4. Why Each Part

| Part | Why |
|---|---|
| `logger` (pino) instead of `console.log` | Pino writes pure JSON — grep-able, parseable, consistent |
| `phase` only on phase events | AC2: log readers shouldn't see `phase: undefined` on unrelated events |
| `outcome` only on terminal events | AC2: same reason — only meaningful at the end |
| `duration_ms` computed from `createdAt` | AC4: measures full lifecycle, not just worker pickup time |
| Logging inside `emitSocketEvent` | One place — impossible to forget a call site |

---

## 5. Tricky Parts

**The `??` operator:**
```ts
// metadata.phase exists?  use it.  otherwise derive from event name.
const phase = (metadata.phase as string | undefined) ?? EVENT_PHASE[event];

// retry passes:   { phase: "phase_1", attempt: 2 }  → uses "phase_1" from metadata
// phase_1_started passes nothing               → uses EVENT_PHASE["phase_1_started"] = "phase_1"
```

**The `&&` spread trick:**
```ts
...(phase !== undefined && { phase })
// if phase = "phase_1"  → spreads { phase: "phase_1" } into log object
// if phase = undefined  → && short-circuits, nothing added
```
This keeps logs clean — no junk `undefined` fields.

## 6. Flow Diagram

```
emitSocketEvent("task-123", "phase_1_started", {})
         │
         ├─ phase = EVENT_PHASE["phase_1_started"] → "phase_1"
         ├─ outcome = not terminal                 → undefined
         │
         ├─ payload = {
         │     task_id:   "task-123",
         │     event:     "phase_1_started",
         │     timestamp: "2026-04-28T...",
         │     metadata:  null
         │  }
         │
         ├─ logger.info → stdout JSON:
         │  {
         │    "task_id":   "task-123",
         │    "event":     "phase_1_started",
         │    "timestamp": "2026-04-28T...",
         │    "metadata":  null,
         │    "phase":     "phase_1"       ← added
         │                                 ← outcome absent (not terminal)
         │  }
         │
         └─ io.to("task:task-123").emit(...)  ← socket clients get payload


emitSocketEvent("task-123", "completed", { duration_ms: 4821 })
         │
         ├─ phase   = undefined (not a phase event)
         ├─ outcome = "completed"
         │
         └─ logger.info → stdout JSON:
            {
              "task_id":   "task-123",
              "event":     "completed",
              "timestamp": "...",
              "metadata":  { "duration_ms": 4821 },
              "outcome":   "completed"    ← added
                                          ← phase absent
            }
```

## 7. Acceptance Criteria Checklist

| AC | Status | How satisfied |
|---|---|---|
| AC1: every log has `task_id`, `event`, `timestamp`, metadata | ✅ | `emitter.ts` — `logger.info` with full payload |
| AC2: `phase` on phase events, `outcome` on terminal only | ✅ | derived + conditionally spread |
| AC3: fallback logs with taskId + trigger + state | ✅ | `reason` in metadata at all fallback call sites |
| AC4: terminal logs with taskId + state + `duration_ms` | ✅ | computed from `createdAt` at terminal call sites |
| AC5: structured JSON, no prose for system events | ✅ | task/job logs use `logger`; startup stays `console.log` |
| AC6: filter by `task_id` as timeline | ✅ | `task_id` on every log → `grep '"task_id":"<uuid>"'` works |
